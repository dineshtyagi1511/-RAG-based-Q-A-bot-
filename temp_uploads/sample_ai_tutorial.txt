CHAPTER 1: WHAT IS ARTIFICIAL INTELLIGENCE?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Artificial Intelligence (AI) is the simulation of human intelligence processes
by computer systems. These processes include learning (acquiring information and
rules for using information), reasoning (using rules to reach approximate or
definite conclusions), and self-correction.

AI was formally founded as a discipline in 1956 at a Dartmouth College conference
where John McCarthy coined the term. Today AI encompasses everything from spam
filters to autonomous vehicles and large language models.

Types of AI by Capability:
- Narrow AI (Weak AI): Systems designed for a specific task, such as voice
  assistants, recommendation systems, and image recognition. All commercially
  deployed AI today falls in this category.
- General AI (AGI): Hypothetical AI that can perform any intellectual task a
  human can. Does not yet exist.
- Superintelligent AI (ASI): Hypothetical AI that surpasses human intelligence
  across all domains. Entirely theoretical.

Types of AI by Approach:
- Rule-based Systems: Follow explicit if-then rules written by experts.
- Machine Learning: Learn patterns from data without being explicitly programmed.
- Deep Learning: Use multi-layered neural networks to learn hierarchical
  representations.
- Generative AI: Create new content — text, images, audio, code — based on
  learned patterns.

CHAPTER 2: MACHINE LEARNING FUNDAMENTALS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Machine Learning (ML) is a subset of AI that provides systems the ability to
automatically learn and improve from experience without being explicitly
programmed. ML focuses on developing algorithms that can access data and use
it to learn for themselves.

The three main types of Machine Learning:

1. Supervised Learning
   The algorithm is trained on labelled data — input-output pairs. It learns
   to map inputs to outputs and generalises to new, unseen data.
   Examples: spam detection, image classification, price prediction.
   Common algorithms: Linear Regression, Logistic Regression, Decision Trees,
   Random Forests, Support Vector Machines, Neural Networks.

2. Unsupervised Learning
   The algorithm is given data with no labels and must discover patterns on its
   own. Common tasks include clustering, dimensionality reduction, and anomaly
   detection.
   Examples: customer segmentation, topic modelling, anomaly detection.
   Common algorithms: K-Means, DBSCAN, PCA, Autoencoders, GANs.

3. Reinforcement Learning
   An agent learns by interacting with an environment, receiving rewards for
   correct actions and penalties for wrong ones. Inspired by behavioural
   psychology.
   Examples: game playing (AlphaGo), robotic control, recommendation systems.
   Key concepts: agent, environment, state, action, reward, policy.

Key ML Concepts:
- Training set: data used to train the model.
- Validation set: data used to tune hyperparameters.
- Test set: held-out data used for final evaluation.
- Overfitting: model memorises training data and fails to generalise.
- Underfitting: model is too simple to capture patterns in data.
- Bias-variance tradeoff: balance between model complexity and generalisation.
- Gradient Descent: optimisation algorithm to minimise loss function.
- Backpropagation: algorithm for computing gradients in neural networks.

CHAPTER 3: DEEP LEARNING AND NEURAL NETWORKS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Deep Learning is a subset of Machine Learning using artificial neural networks
with many layers (hence "deep"). These networks are inspired by the structure
and function of the human brain.

Neural Network Architecture:
- Input Layer: receives raw data features.
- Hidden Layers: intermediate layers that learn representations. "Deep" refers
  to having many hidden layers.
- Output Layer: produces the final prediction.
- Neurons (nodes): basic computation units that apply a weighted sum + activation.
- Weights and Biases: learnable parameters adjusted during training.
- Activation Functions: introduce non-linearity. Common ones: ReLU, Sigmoid,
  Tanh, Softmax.

Popular Deep Learning Architectures:

Convolutional Neural Networks (CNN):
  Designed for grid-like data such as images. Use convolution layers to detect
  local patterns. Applications: image classification, object detection, medical
  imaging.

Recurrent Neural Networks (RNN) / LSTM:
  Designed for sequential data. LSTMs (Long Short-Term Memory) solve the
  vanishing gradient problem. Applications: time-series, speech recognition,
  language modelling.

Transformers:
  Introduced in "Attention is All You Need" (Vaswani et al., 2017). Use
  self-attention mechanisms to model relationships between all tokens regardless
  of distance. Foundation of modern LLMs.
  Key components: multi-head attention, positional encoding, feed-forward layers.
  Applications: NLP, computer vision (ViT), audio (Whisper).

Generative Adversarial Networks (GAN):
  Two networks — a Generator and a Discriminator — compete in a zero-sum game.
  Applications: image synthesis, style transfer, data augmentation.

Diffusion Models:
  Learn to reverse a noise-adding process. State-of-the-art for image
  generation. Examples: Stable Diffusion, DALL-E, Midjourney.

CHAPTER 4: LARGE LANGUAGE MODELS (LLMs)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Large Language Models are transformer-based models trained on massive text
corpora to predict the next token in a sequence. Through this simple objective
they acquire broad world knowledge and language understanding.

Key LLMs:
- GPT-4 / GPT-4o (OpenAI): Multimodal, state-of-the-art reasoning.
- GPT-4o-mini (OpenAI): Smaller, faster, cost-efficient version.
- Claude 3 (Anthropic): Focus on safety and long context (200K tokens).
- Gemini (Google): Multimodal, trained on Google's data.
- LLaMA 3 (Meta): Open-weight model, widely used for fine-tuning.
- Mistral / Mixtral: Efficient open-source models with strong performance.

LLM Capabilities:
- Text generation, summarisation, translation, Q&A.
- Code generation and debugging.
- Instruction following and chat.
- Reasoning, maths (with chain-of-thought prompting).

LLM Limitations:
- Hallucination: confidently generating false facts.
- Knowledge cutoff: no information after training date.
- Context length limits (though growing rapidly).
- Lack of real-time information.
- Sensitive to prompt wording.

Retrieval-Augmented Generation (RAG) is the primary technique to address
hallucination and knowledge cutoff limitations.

CHAPTER 5: RETRIEVAL-AUGMENTED GENERATION (RAG)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

RAG (Retrieval-Augmented Generation) is an AI framework that combines a
retrieval system with a generative model. Instead of relying solely on the
knowledge baked into model weights during training, RAG fetches relevant
external documents at inference time and provides them to the LLM as context.

Why RAG?
- Reduces hallucinations by grounding answers in retrieved facts.
- Overcomes knowledge cutoff: documents can be updated without retraining.
- Provides source attribution for answers.
- Cheaper than fine-tuning for adding domain knowledge.
- Allows personalisation with private/proprietary documents.

Standard RAG Pipeline:

Step 1 — Ingestion (Offline):
  Load documents → Split into chunks → Embed each chunk → Store in vector DB.

Step 2 — Retrieval (Online):
  Embed user query → Search vector DB for top-k similar chunks.

Step 3 — Generation (Online):
  Inject retrieved chunks into prompt → LLM generates grounded answer.

Advanced RAG Techniques:
- HyDE (Hypothetical Document Embeddings): generate a hypothetical answer,
  embed it, and use that for retrieval.
- Self-RAG: LLM decides when to retrieve and critiques its own output.
- Corrective RAG (CRAG): evaluates retrieved documents and falls back to web
  search if quality is low.
- RAG Fusion: generate multiple search queries, retrieve for each, and fuse.
- Graph RAG: combine vector search with a knowledge graph for structured
  reasoning.
- Multi-hop RAG: iterative retrieval for complex multi-step questions.

RAG Evaluation Metrics:
- Faithfulness: is the answer supported by the context?
- Answer Relevance: does the answer address the question?
- Context Precision: what fraction of retrieved chunks are relevant?
- Context Recall: were all needed chunks retrieved?
Tools: RAGAS, TruLens, LangSmith.

CHAPTER 6: VECTOR DATABASES AND EMBEDDINGS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Embeddings are dense vector representations of text (or images, audio) that
capture semantic meaning. Semantically similar content has nearby vectors in
high-dimensional space.

Popular Embedding Models:
- text-embedding-3-small (OpenAI): 1536 dims, fast and cost-effective.
- text-embedding-3-large (OpenAI): 3072 dims, higher quality.
- text-embedding-ada-002 (OpenAI): 1536 dims, legacy model.
- all-MiniLM-L6-v2 (HuggingFace): 384 dims, open-source, fast on CPU.
- BAAI/bge-large-en (HuggingFace): 1024 dims, strong open-source option.

Vector Databases:
Vector databases store embeddings and support approximate nearest-neighbour
(ANN) search efficiently.

Qdrant:
  Open-source, written in Rust. Supports payload filtering, hybrid search
  (dense + sparse), named vectors, and built-in quantisation. Available as
  on-prem or Qdrant Cloud. Recommended for production RAG pipelines.
  Key features: HNSW index, scalar/product quantisation, REST and gRPC API.

Pinecone:
  Fully managed, serverless. Easy to start but can become expensive at scale.

Weaviate:
  Open-source, supports multi-modal data and built-in classification.

Chroma:
  Lightweight, embeds in Python process. Good for prototyping.

FAISS:
  Facebook AI Similarity Search. Library (not a database). Fast but in-memory
  only; no persistence without extra code.

pgvector:
  PostgreSQL extension. Good if you're already on Postgres and want vector
  search without a separate service.

Similarity Metrics:
- Cosine Similarity: angle between vectors; most common for text.
- Dot Product: similar to cosine when vectors are normalised.
- Euclidean (L2) Distance: absolute distance; less common for text.

Indexing Algorithms:
- HNSW (Hierarchical Navigable Small World): best recall/speed tradeoff.
  Used by Qdrant and most modern vector DBs.
- IVF (Inverted File Index): good for very large datasets.
- Flat (Exact Search): exhaustive; accurate but slow at scale.

CHAPTER 7: LANGCHAIN FRAMEWORK
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

LangChain is an open-source framework for building LLM-powered applications.
It provides abstractions for chaining together components: LLMs, prompts,
memory, tools, and retrievers.

Core LangChain Components:

LLMs and Chat Models:
  Wrappers around provider APIs (OpenAI, Anthropic, Google, Hugging Face).
  ChatOpenAI, ChatAnthropic, ChatGoogleGenerativeAI, etc.

Prompts:
  PromptTemplate, ChatPromptTemplate, SystemMessage, HumanMessage.
  Support for few-shot examples, partial formatting, and input validation.

Chains (LCEL — LangChain Expression Language):
  Compose components with the pipe operator (|).
  Example: retriever | format_docs | prompt | llm | StrOutputParser()
  Benefits: streaming, batching, async, tracing out of the box.

Retrievers:
  Abstraction over vector stores. Support similarity, MMR (Maximal Marginal
  Relevance), and multi-query retrieval.

Memory:
  ConversationBufferMemory, ConversationSummaryMemory, etc. for multi-turn chat.

Tools and Agents:
  LangChain agents use LLMs to decide which tools to call (web search, code
  execution, APIs). Integrated with LangGraph for stateful workflows.

LangGraph:
  Extension for building stateful, multi-actor graph-based workflows. Supports
  loops, branching, and persistent state. Ideal for agentic RAG and copilots.

LangSmith:
  Observability platform for LangChain apps. Trace every LLM call, evaluate
  outputs, and run regression tests.

CHAPTER 8: PROMPT ENGINEERING
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Prompt engineering is the practice of designing inputs to LLMs to obtain
desired outputs. It is a critical skill for building reliable AI applications.

Core Techniques:

Zero-Shot Prompting:
  Ask the model directly without examples.
  "Classify the sentiment of: 'The product is great!'"

Few-Shot Prompting:
  Provide 2–5 examples of input-output pairs before the actual query.
  Significantly improves performance on classification and formatting tasks.

Chain-of-Thought (CoT):
  Ask the model to reason step-by-step before giving the final answer.
  "Let's think step by step." increases accuracy on reasoning tasks.

System Prompts:
  Instructions given in the system role to set behaviour, persona, and
  constraints. Never visible to users in production apps.

Role Prompting:
  "You are an expert financial analyst…" primes the model with domain context.

Structured Output Prompting:
  Ask for JSON, XML, or specific formats. Use with response_format={"type":
  "json_object"} in OpenAI API for reliable structured outputs.

Temperature and Sampling:
  Temperature = 0: deterministic, best for factual Q&A.
  Temperature = 0.7–1.0: more creative, good for brainstorming.
  Top-p (nucleus sampling): controls vocabulary diversity.

Context Window Management:
  Every LLM has a maximum context window (e.g., 128K for GPT-4o).
  Strategies: summarise old turns, truncate, or use sliding window.

Guardrails:
  System prompts that restrict the model to a topic domain.
  "Answer ONLY using the provided context. Do not use prior knowledge."

CHAPTER 9: AI SAFETY AND ETHICS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

AI Safety refers to the field of ensuring AI systems behave as intended and
remain beneficial as they become more capable.

Key Safety Concerns:

Alignment:
  Ensuring AI goals and values align with human intentions. Misaligned AI may
  optimise for proxy metrics that diverge from true human values.

Robustness:
  AI systems should perform reliably under distribution shift, adversarial
  inputs, and edge cases.

Interpretability (Explainability):
  Understanding why a model makes a specific decision. SHAP, LIME, and
  mechanistic interpretability research address this.

Bias and Fairness:
  Models trained on biased data perpetuate and amplify societal biases. Auditing
  datasets and model outputs is essential before deployment.

Privacy:
  Models can memorise and leak training data. Differential privacy and
  federated learning are mitigation strategies.

Prompt Injection:
  Malicious content in retrieved documents or user input that hijacks LLM
  behaviour. Critical concern in RAG systems. Mitigated by output validation,
  sandboxing, and tools like LLM Guard.

LLM Guard:
  Open-source library for input/output scanning of LLM applications.
  Input scanners: PromptInjection, Toxicity, BanTopics, TokenLimit, PII.
  Output scanners: Relevance, NoRefusal, Bias, Sensitive.

OWASP Top 10 for LLM Applications:
  1. Prompt Injection
  2. Insecure Output Handling
  3. Training Data Poisoning
  4. Model Denial of Service
  5. Supply Chain Vulnerabilities
  6. Sensitive Information Disclosure
  7. Insecure Plugin Design
  8. Excessive Agency
  9. Overreliance
  10. Model Theft

Responsible AI Principles:
  - Fairness: do not discriminate based on protected characteristics.
  - Reliability: behave consistently and safely.
  - Privacy: protect user data.
  - Inclusiveness: serve diverse users.
  - Transparency: be clear about AI capabilities and limitations.
  - Accountability: maintain human oversight.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

MLOps is the practice of streamlining the ML lifecycle from development to
production deployment and monitoring.

Key MLOps Components:

Version Control:
  Git for code. DVC (Data Version Control) for datasets and model artefacts.

Experiment Tracking:
  MLflow, Weights & Biases (W&B), Neptune — log metrics, hyperparameters,
  and artefacts across experiments.

Model Registry:
  MLflow Model Registry, Hugging Face Hub — store, version, and stage models.

CI/CD for ML:
  GitHub Actions, GitLab CI — automate testing, training, evaluation, and
  deployment pipelines. Key stages: lint → test → train → evaluate → deploy.

Containerisation:
  Docker for packaging models and dependencies. Key: CPU-only builds for
  inference-only containers to reduce size and cost.

Orchestration:
  AWS ECS Fargate: serverless containers; no cluster management required.
  Good for stateful long-running services (LangGraph, llm-guard model loading).
  Kubernetes: full orchestration for large-scale ML systems.

Serving:
  FastAPI: fast, async, OpenAPI-native REST framework for ML APIs.
  Triton Inference Server: NVIDIA, for GPU inference at scale.
  Ray Serve: distributed serving with Python.

Monitoring:
  Data drift detection: Evidently AI, Whylogs.
  Model performance: track accuracy, latency, and error rates in production.
  LLM-specific: RAGAS for RAG quality, LangSmith for tracing.

Secrets Management:
  AWS SSM Parameter Store, AWS Secrets Manager, HashiCorp Vault.
  Never embed secrets in code or Docker images.

Redis in Production AI:
  Redis is used as: semantic cache for LLM calls (reduce cost and latency),
  rate limiting store (sliding window counters), session state for chatbots,
  and message broker for async inference queues.

AI & MACHINE LEARNING: A PRACTICAL TUTORIAL
=============================================

This document is a demo knowledge base for the RAG Q&A Bot. It covers
machine learning fundamentals, retrieval-augmented generation, embeddings,
vector search, and AI application security.


1. SUPERVISED VS UNSUPERVISED LEARNING
---------------------------------------

Supervised learning trains a model on labeled data, meaning each training
example comes with a known correct answer. The model learns a mapping from
inputs to outputs and is evaluated by how well it predicts labels on unseen
data. Common tasks include classification (e.g. spam vs not spam, image
categories) and regression (e.g. predicting house prices, stock returns).
Typical algorithms include linear and logistic regression, decision trees,
random forests, gradient boosting (XGBoost, LightGBM), support vector
machines, and neural networks trained with backpropagation against a
labeled target.

Unsupervised learning works with unlabeled data, where the goal is to
discover structure or patterns rather than predict a known answer. There is
no "correct" output to compare against during training. Common tasks
include clustering (grouping similar data points, e.g. k-means, DBSCAN,
hierarchical clustering), dimensionality reduction (compressing data while
preserving structure, e.g. PCA, t-SNE, UMAP), and anomaly detection.

The key distinction is the presence or absence of labeled targets. Semi-
supervised learning sits between the two, using a small amount of labeled
data alongside a larger pool of unlabeled data. Self-supervised learning,
which underlies most modern LLM pretraining, creates its own labels from
the input data itself (e.g. predicting the next token in a sequence),
blurring the line further but is generally grouped with unsupervised
methods since no human-provided labels are required.


2. HOW RETRIEVAL-AUGMENTED GENERATION (RAG) REDUCES HALLUCINATIONS
--------------------------------------------------------------------

LLMs generate text by predicting the most probable next token based on
patterns learned during training. When a model is asked about something
outside its training data, recent events, private documents, or niche
facts, it can produce fluent but factually incorrect text. This is called
hallucination, and it happens because the model is optimizing for
plausibility, not for verified truth, and it has no built-in way to check
its own outputs against a ground-truth source.

Retrieval-Augmented Generation addresses this by giving the model relevant
external context at inference time, rather than relying purely on what it
memorized during training. A typical RAG pipeline works like this: a
user's query is converted into an embedding vector, that vector is used to
search a vector database for the most semantically similar chunks of
trusted source documents, and those chunks are inserted into the prompt
alongside the original question. The LLM then generates its answer
grounded in that retrieved context, often citing which chunk supports which
claim.

This reduces hallucinations in two main ways. First, the model has direct,
verifiable source text in front of it, so it can quote or paraphrase
instead of guessing from memory. Second, because RAG systems can return
"I don't know" or refuse to answer when no relevant context is found,
gaps in knowledge become visible instead of being silently filled in with
fabricated text. RAG does not eliminate hallucination entirely; a model can
still misread or contradict the retrieved context, and weak retrieval
(missing or irrelevant chunks) leads to weak grounding. Techniques like
HyDE (Hypothetical Document Embeddings), reranking, and self-reflective
RAG (where the model critiques its own draft answer against the source
before finalizing it) are used to further reduce these failure modes.


3. OPENAI EMBEDDING MODELS AND THEIR DIMENSIONS
--------------------------------------------------

An embedding model converts text into a fixed-length vector of numbers
that captures its semantic meaning, such that similar pieces of text end
up close together in vector space. OpenAI currently offers three embedding
models:

- text-embedding-3-small: 1536 dimensions. Optimized for cost and speed
  while still offering strong retrieval performance. This is the default
  choice for most RAG applications, including this bot.
- text-embedding-3-large: 3072 dimensions. Higher accuracy on retrieval
  and semantic similarity benchmarks, at higher cost and storage overhead.
  Useful when retrieval quality is the bottleneck.
- text-embedding-ada-002 (legacy): 1536 dimensions. Superseded by the v3
  models, which outperform it at a similar or lower price point.

Both v3 models support a "dimensions" parameter that lets you shorten the
output vector (for example, requesting 1536 dimensions from the 3072-
dimension large model) at a small accuracy cost, which is useful for
reducing vector database storage and search latency without switching
models entirely. Choosing an embedding model is a trade-off between
retrieval quality, vector storage cost, and query latency — smaller
dimensions mean less storage and faster nearest-neighbor search, but can
lose some nuance in highly technical or ambiguous text.


4. HNSW AND VECTOR DATABASES
--------------------------------

HNSW (Hierarchical Navigable Small World) is a graph-based algorithm for
approximate nearest neighbor (ANN) search. Instead of comparing a query
vector against every stored vector (which becomes too slow at scale),
HNSW organizes vectors into multiple layers of graphs. The top layer has
few nodes with long-range connections for fast coarse navigation; lower
layers have progressively more nodes with shorter connections for fine-
grained search. A search starts at the top layer and "zooms in" layer by
layer, quickly narrowing down to a small candidate set of nearest
neighbors without scanning the whole dataset.

This makes HNSW extremely fast for similarity search even over millions
of vectors, at the cost of being approximate rather than exact — it may
occasionally miss the true nearest neighbor in exchange for large speed
gains. The trade-off is tunable: increasing the graph's connectivity and
search breadth improves recall (accuracy) at the cost of speed and memory.

Several vector databases use HNSW as their core indexing method, including
Qdrant (used in this project), Weaviate, and Milvus. Pinecone uses its own
proprietary indexing approach, and Postgres's pgvector extension supports
both HNSW and the older IVFFlat method. Beyond indexing, vector databases
typically add features RAG applications rely on: metadata filtering
(restricting search to documents matching certain tags), hybrid search
(combining vector similarity with traditional keyword search), and
payload storage so each vector can carry its original text and source
metadata alongside it.


5. OWASP TOP 10 FOR LLM APPLICATIONS (2025)
-----------------------------------------------

The OWASP Top 10 for LLM Applications is a community-maintained list of
the most critical security risks in generative AI systems, first
published in 2023 and substantially revised for 2025 to reflect the rise
of RAG pipelines and autonomous agents. The current ten categories are:

LLM01:2025 Prompt Injection — attackers craft input that overrides the
  system's intended instructions, either directly in the user's message
  or indirectly hidden inside a document, webpage, or email the model
  later processes.
LLM02:2025 Sensitive Information Disclosure — the model leaks private,
  proprietary, or regulated data through its outputs, logs, or cached
  responses.
LLM03:2025 Supply Chain — vulnerabilities introduced through third-party
  models, datasets, fine-tuning adapters, or libraries the application
  depends on.
LLM04:2025 Data and Model Poisoning — tampered training or fine-tuning
  data introduces hidden backdoors or biased behavior.
LLM05:2025 Improper Output Handling — insufficient validation or
  sanitization of model output before it's used downstream (e.g. executed
  as code or rendered as HTML), enabling injection-style attacks.
LLM06:2025 Excessive Agency — an LLM-based agent is granted more tools,
  permissions, or autonomy than it needs, increasing the blast radius of
  any mistake or manipulation.
LLM07:2025 System Prompt Leakage — system prompts are exposed, which
  matters because system prompts should never be treated as a security
  boundary in the first place.
LLM08:2025 Vector and Embedding Weaknesses — risks specific to RAG
  systems, such as poisoned or manipulated content being embedded and
  retrieved as if it were trustworthy context, or sensitive data leaking
  through stored vectors.
LLM09:2025 Misinformation — the model produces confident, plausible-
  sounding but false output that users act on without verification.
LLM10:2025 Unbounded Consumption — uncontrolled or excessive resource
  usage (queries, tokens, compute) leads to denial of service or runaway
  cost, sometimes called "denial of wallet."

For a RAG application like this one, Prompt Injection, Sensitive
Information Disclosure, Vector and Embedding Weaknesses, and Unbounded
Consumption are the most directly relevant categories, which is why this
project layers input sanitization, rate limiting, and ML-based scanning on
top of the core retrieval pipeline.


6. LLM GUARD AND PROMPT INJECTION PROTECTION
------------------------------------------------

LLM Guard is an open-source toolkit of input and output "scanners" that
sit between the user, the LLM, and the application, inspecting text before
it's processed and after it's generated. Rather than relying on the LLM
itself to police its own inputs and outputs, LLM Guard uses dedicated
classifier models and rule-based checks for specific risks.

On the input side, common scanners include PromptInjection (a classifier
trained to detect text that resembles known jailbreak or instruction-
override patterns), Toxicity (flags harmful or abusive language), and
TokenLimit (rejects inputs that are unreasonably long, which both controls
cost and reduces the surface area for hidden injected instructions). On
the output side, a Relevance scanner checks whether the model's answer
actually relates to the retrieved context and the user's question, which
helps catch cases where the model has drifted off-topic or been steered by
an injected instruction.

Prompt injection detection is inherently probabilistic, not a guarantee.
A classifier can be evaded by novel phrasing it wasn't trained on, so LLM
Guard is best understood as one layer in a defense-in-depth strategy.
Combining it with simpler, always-on protections, regex-based input
sanitization, strict separation of system instructions from user and
retrieved content, rate limiting, and output validation, narrows the
attack surface even when any single layer can be bypassed.


7. THE ROLE OF REDIS IN PRODUCTION AI SYSTEMS
--------------------------------------------------

Redis is an in-memory data store frequently used as supporting
infrastructure in production LLM applications, mainly because LLM calls
are slow and expensive compared to a cache lookup. Common uses include:

- Response caching: hashing a normalized query (e.g. with SHA-256) and
  storing the LLM's answer against that hash, so an identical or
  near-identical question can be served instantly from cache instead of
  re-running retrieval and generation. A time-to-live (TTL) keeps cached
  answers from going stale indefinitely.
- Semantic caching: a variant where cache lookups are based on embedding
  similarity rather than exact text match, so paraphrased questions can
  still hit the cache.
- Rate limiting: Redis's atomic counters and built-in expiry make it a
  natural fit for sliding-window or token-bucket rate limiters that throttle
  how many requests a user can make in a given time window, directly
  mitigating Unbounded Consumption risk.
- Session and conversation state: storing recent chat history or
  multi-turn context so a stateless API server can maintain continuity
  across requests without hitting a slower primary database.
- Vector search: modern Redis (via the RediSearch module) can also store
  and query vector embeddings directly, which lets some teams use a single
  Redis deployment for both caching and retrieval, though dedicated vector
  databases like Qdrant typically offer more specialized indexing options
  for large-scale RAG workloads.

Because Redis operates in memory, it adds minimal latency to the request
path while meaningfully reducing both response time and the cost of
repeated LLM and embedding API calls, making it a near-default component
in production-grade RAG architectures.
